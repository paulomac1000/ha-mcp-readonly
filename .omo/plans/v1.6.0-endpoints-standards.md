### Interview Summary
**Key Discussions**:
- N1/N3/N6 excluded: hardcoded entity IDs for one specific household — not suitable for general-purpose MCP server
- M5 skipped: `category` filter already exists on `search_automations` (line 2128)
- M1 replaced: `get_automation_traces` cannot work without WebSocket; trace-like functionality added to `get_automation_usage_stats` via REST API logbook + history
- M3 convention: comma-separated strings (matches all 12 existing batch tools; VS Code MCP Issue #277462 confirms array params cause client bugs)
- N2 as extension: enhance `get_entity_dependencies` with deeper search, not a new tool
- N4 universalized: `compare_templates` — generic template comparison, not tied to any macro file
- MCP standards: enforce `entity_ids: str` (comma-separated) for all batch tools; document `detail_level`/`compact`/`include_*` patterns

**Research Findings**:
- HA traces are WebSocket-only (`trace/list`, `trace/get`); HA officially deprecated new REST endpoints (discussion #1074)
- Logbook REST API (`/api/logbook/{datetime}?entity=xxx`) provides trigger events, context IDs, state changes
- FastMCP + OpenAPI: array params serialized as comma-separated for interop
- `get_entity_details` (storage.py) lacks `compact`; `get_entity_state` (states.py) has basic `compact` but no domain-specific attribute retention
